{{- /* Generic redirect layout. Used by: (a) auto-generated short URLs from data/redirects.yaml (b) any page that sets `layout: redirect` and a `target_url:` param. Build-time rewriting -------------------- For INTERNAL targets (paths starting with "/"), a post-build step (scripts/apply_rewrites.py) copies the fully rendered target page over this redirect stub so the visitor sees the real content at the short URL with no redirect. This meta-refresh page therefore only survives in practice for EXTERNAL targets, or as a fallback if the post-build rewrite is skipped (e.g. local dev builds). Redirect "status code" semantics on a static host -------------------------------------------------- GitHub Pages cannot send a real HTTP 3xx status, so the redirect is always implemented as a meta-refresh page that returns 200. To still honour the chosen status semantically, we change what we tell crawlers: 301 / 308 (permanent) → canonical points to the TARGET so search engines transfer page authority. We also advise crawlers not to index the shim. 302 / 307 (temporary) → canonical points back to the CURRENT URL (i.e. self), so search engines do not transfer authority to the target. Still noindex (we never want the shim itself ranking) but `follow` so links are crawled. A Netlify-format `static/_redirects` file is also generated alongside these stubs, so a future migration to a host that honours `_redirects` picks up the real HTTP status code automatically. Meta-refresh is universally supported; we intentionally skip a JS fallback because Hugo's HTML minifier mangles string literals inside inline
